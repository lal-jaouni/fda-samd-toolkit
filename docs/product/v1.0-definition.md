# v1.0 Definition: Stable Toolkit with Worked Examples for 3 Device Classes

This document defines the explicit success criteria for the FDA SaMD Toolkit to reach v1.0 stability.

v1.0 is the first release where the toolkit is positioned as a stable, production-ready foundation for FDA submissions. It is NOT the end of development; it is the beginning of backward-compatibility guarantees and broader commercial adoption.

---

## v1.0 North-Star Metric

**3+ real FDA 510(k) submissions cite the FDA SaMD Toolkit in their design history files by release date (January 2027).**

This proves product-market fit and demonstrates that the toolkit solves real regulatory problems for real companies.

---

## Success Criteria

### 1. Three Stable Device Classes with Worked Examples

A "stable" device class means the toolkit has a complete, validated end-to-end example showing all components integrated for that modality. Users can copy, customize, and submit.

#### Device Class 1: Cardiac AI (ECG-based detection)

**Current state**: CardioGuard worked example ships with v0.1 (fictional ECG arrhythmia detector).

**v1.0 requirement**: CardioGuard completed and validated against a real cleared device public submission.

**Deliverables**:
- Worked example: `examples/cardioguard-ecg-ai/` (expanded from v0.1)
  - `pccp.yaml`: PCCP for future ECG algorithm updates
  - `model_card.yaml`: Training data, validation results, subpopulation performance
  - `validation_protocol.yaml`: Clinical validation for ECG signals
  - `bias_evaluation.yaml`: Demographic performance breakdowns
  - `monitoring_plan.yaml`: Real-world performance monitoring
  - `sbom.yaml`: Software component inventory
  - `checklist.yaml`: 58-item submission readiness checklist
  - Generated outputs: PCCP markdown, model card, validation report, bias report, monitoring plan, SBOM JSON, checklist HTML
  - README: Step-by-step guide to customize for your ECG device
- Blog post / case study: "How We Built CardioGuard: A 510(k) ECG-AI Submission from Start to Finish"
- Validation: Dissected against Anumana ECG-AI (cleared March 2026, public 510(k) summary) to confirm structure matches real submissions

**Rationale**: Cardiac/biosignal AI is strongest use case (Tier 1 consulting candidates: Cardiosense, Salomatic, Anumana, Eko). Public reference submission (Anumana) provides validation target.

**Definition of done**:
- All 7 YAML configs present and generate valid outputs
- Example generator code runs without errors
- Blog post published on toolkit website or Medium
- Case study linked in README

### 5. External Contribution

At least 1 external contributor (not the original author) has submitted and merged a pull request.

### 6. Real Submission Citation (North-Star Metric)

At least 1 real FDA 510(k) submission explicitly cites the toolkit in the design history file or submission package.

**Rationale**: Proves real product-market fit and solves genuine regulatory pain.

**Definition of done**: Company provides (with permission) evidence that their submission references the toolkit (email, LinkedIn post, blog, or public FDA document).

### 7. Semver API Stability Guarantee

After v1.0.0 releases:
- **No breaking changes to CLI subcommands** without MAJOR version bump (e.g., renaming `fda-samd pccp` -> `fda-samd precp` requires v2.0)
- **No breaking changes to Python public API** without MAJOR version bump (e.g., class renames, removed functions)
- **YAML schema changes** are allowed with deprecation warnings through v1.x (removal requires v2.0)

**Definition of done**: CONTRIBUTING.md and CHANGELOG.md document semver policy clearly.

## Non-Goals for v1.0

Explicitly NOT attempting:

1. **SaaS platform**: Toolkit remains open-source CLI + Python library. No web UI, cloud hosting, or subscription model.
2. **Hosted service**: Users run toolkit locally; no server component.
3. **Telemetry**: No usage tracking, no metrics collection, no phone-home behavior.
4. **EU MDR or CE marking**: Toolkit is US FDA-specific through v1.0. (EU toolkit may be separate project post-v1.0.)
5. **PMA submissions**: Toolkit focuses on 510(k) pathway. PMA is much larger and deferred to v2.0+.
6. **De Novo submissions**: De Novo requires novel classification (different predicate logic). Deferred to v2.0+.
7. **IVD (In Vitro Diagnostics)**: IVDs have separate FDA pathway. Deferred to separate IVD toolkit.
8. **Automated FDA form filling**: Users manually upload eSTAR package to FDA eSubmitter. Form completion is user responsibility.
9. **Legal or regulatory advice**: Toolkit is documentation scaffolding only. Users must work with regulatory professionals.

## Device Class Selection Rationale

### Cardiac AI (ECG-Based)

- **Current**: CardioGuard worked example already in v0.1
- **Market**: Tier 1 consulting candidates (Cardiosense, Anumana, Eko, Salomatic) focus on cardiac AI
- **Public reference**: Anumana ECG-AI cleared March/April 2026 with public 510(k) summary (K232488 or similar)
- **Validation infrastructure**: ECG signal processing well-understood; subpopulation performance straightforward to demonstrate
- **Prior work**: Laith has prior experience with cardiac biosignal analysis (msai-phase3, biosignal-samd-practice repos)

### Medical Imaging AI (X-ray or CT)

- **Market**: Second-most-common AI device class; 25-30% of FDA-cleared AI devices are imaging
- **Public reference**: Real examples available (K232613 Body Check CT via Innolitics, multiple X-ray AI devices)
- **Modality difference**: Imaging validation is different from signals (sensitivity/specificity for classification on images, not time-series)
- **Subpopulation analysis**: Can demonstrate performance across demographics, lesion types, anatomical regions
- **Expansion value**: Shows toolkit works beyond signal processing (cardiac)

### NLP/EHR AI (Clinical Decision Support)

- **Market**: Emerging category; growing number of FDA clearances for EHR AI, NLP pipelines
- **Challenge level**: NLP has unique validation challenges (text interpretation, fairness across populations, domain shift)
- **Public reference**: Some EHR AI devices available for dissection (e.g., sepsis prediction, readmission risk)
- **Differentiation**: Demonstrates toolkit handles diverse data types (time-series, images, text)
- **Expansion value**: Broadest modality coverage = most credible toolkit

Together, these three classes cover the spectrum of FDA AI clearances and show the toolkit works for any AI-enabled device.

## Validation Against Real Submissions

For each worked example, the toolkit maintainer will:

1. **Identify public cleared device**: FDA 510(k) summary for cardiac/imaging/NLP device with public documentation available
2. **Extract submission structure**: Document how the cleared device organized its PCCP, model card, validation protocol, bias analysis, etc.
3. **Map to toolkit components**: Show how CardioGuard/Imaging/NLP example maps to the real submission structure
4. **Generate outputs**: Run toolkit generators and compare to real submission sections
5. **Note gaps**: Document any FDA expectations not yet covered by toolkit (for future work)
6. **Publish case study**: Blog post linking toolkit artifacts to real submission example

**Example**: CardioGuard v1.0 validation

- Cleared device: Anumana ECG-AI for cardiac amyloidosis detection (K-number TBD, cleared March/April 2026)
- Submission structure: 510(k) summary publicly available (Innolitics, FDA website)
- Toolkit mapping: CardioGuard PCCP generator produces similar PCCP structure to Anumana submission
- Output comparison: Generated bias report matches format expected by FDA Jan 2025 guidance
- Case study: "Building CardioGuard: How Our Toolkit Structures Match Anumana's FDA-Cleared Submission"
